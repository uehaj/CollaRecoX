文字起こし → 共同校正パイプライン ロジック仕様

interim → 機械分割（ヒューリスティクス）→ AI補正（結合）→ 確定。2026-06-13 時点の実装。
対象: src/app/realtime/RealtimeClient.tsx（認識画面・録音者）／server.js（中継＋自動校正）／src/app/editor/[sessionId]/CollaborativeEditorV2.tsx（校正画面）。

大前提：音声認識はChromeオンデバイスの Web Speech API（processLocally, ja-JP）。音声は外部送信されない。 認識結果は録音者が単一の WebSocket /api/realtime-ws 経由でサーバへ送り、サーバが Yjs/Hocuspocus 共有ドキュメントへ反映。 校正画面の参加者はその共有ドキュメントを編集する。OpenAI は文字起こしには使わない（自動校正・AI編集の chat completions のみ使用）。

1. テキストの4段階と色

1つの発話テキストは、次の4段階を経て「動かない確定文」になる。校正画面・認識画面で色が進む。

①
interim（グレー） — オンデバイス認識の途中仮説。後続音声で書き換わる（バックトラック）。改行は入れない。
②
機械分割済み（緑） — オンデバイスが確定(final)した生テキスト。サーバの rawBuffer に蓄積。表示用に機械的な改行を細かく入れる（後述）。まだ AI 補正前。
③
AI補正（緑→黒へ） — 緑のうち「最後の改行まで」を AI が処理：誤って割れた一文の結合＋誤字・句読点。新たな分割はしない。
④
確定（黒） — AI補正の結果を共有ドキュメントへ段落として追記のみ（置き換えなし＝人の校正と競合しない）。

2. データフロー

[録音者ブラウザ] オンデバイス認識(Web Speech)

   │  final毎               → local_transcription {text, continuation, gapMs}
   ▼  (WebSocket /api/realtime-ws)
[server.js / 接続ハンドラ]
   ├ rawBuffer に final を蓄積（gapMs≥1200ms かつ非continuation の位置に「ポーズ境界」を記録）
   ├ setPendingText(): 共有docの statusMap に
   │     pendingText     = rawBuffer + interim
   │     pendingGreenLen = rawBuffer.length            (緑/グレーの境界)
   │     pendingBreaks   = 機械改行オフセット(緑のみ・40字基準)  ← そとづけの区切り情報
   └ maybeAutoProofread()（最短5秒間隔／6秒タイマー）
         target = rawBuffer の「最後の機械改行(120字基準)まで」  ← それ以降(ライブ末尾)は緑のまま

         ② AI(proofreadText, gpt-4.1-mini)で「結合＋誤字＋句読点」

   ▼  (Yjs同期)
[校正画面 CollaborativeEditorV2 / 録音画面 RealtimeClient]
   黒 = 共有doc段落(AI確定)  緑 = rawBuffer(AI待ち)  グレー = interim

3. 機械分割（ヒューリスティクス）

mechanicalBreakOffsets(text, pauseOffsets, maxLen) が「改行を入れるべきオフセット配列」を返す。AIには頼らない。

  • 一次：無音ポーズ境界 — クライアントが local_transcription に添える gapMs（前確定からの無音ms）が PAUSE_BREAK_MS=1200 以上の位置で段落区切り。
  • 二次：字数＋語境界 — 1セグメントが maxLen を超えたら、Intl.Segmenter('ja',{granularity:'word'}) の語境界（句読点があれば句読点直後を優先）で分割。語の途中では切らない。
用途定数意図
緑の表示改行GREEN_DISPLAY_MAX = 40緑を短い行に細かく刻む。行が伸びて再ワラップ→視線が泳ぐのを防ぐ（読みの安定）。AIが後で結合する前提なので細かくてよい。
確定(AI補正)単位MECH_MAX_LINE = 120AIに渡す1セグメントの上限。大きめにして文の途中分割を減らし、AIの結合品質を保つ。

4. AI補正（結合＋誤字＋句読点）

proofreadText() が gpt-4.1-mini に投げる。分割はしない・結合と校正だけに特化したプロンプト：

あなたは日本語の会議文字起こしを整える校正アシスタントです。
【校正対象】には機械処理で改行が挿入されています（位置は不正確で、一文の途中で割れていることがあります）。

最優先（改行の扱い）:
 - 改行は段落区切り。原則維持する。
 - 結合してよいのは「前の行が文の途中で終わり、次の行がその続き」＝一文が途中で割れている場合だけ。
 - 各行が文として成立していれば、その改行は絶対に維持する（完結文どうしを1段落にまとめない）。
 - 新たな改行（分割）は追加しない。
その他: 誤字・誤変換の修正／句読点(、。)の補完／内容や文体は変えない／
        末尾が文の途中なら勝手に完結させない（続きは次回送られる）。
【文脈】= 直前までに確定した校正テキスト末尾（参考のみ・出力しない）。

5. 確定の境界とライブ末尾

  • AI補正・確定の対象は rawBuffer の「最後の機械改行(120字基準)まで」に限定する。
  • 最後の改行より後＝ライブ末尾（今まさに話している部分）は、AI（結合・修正）にも確定にもかけず、緑のまま残す。人が見ている末尾が大きく動いて見失うのを防ぐため。
  • 各確定バッチは必ず改行境界で終わるので、追記時は forceNewParagraph=true（既存内容がある場合）で必ず新段落として追記し、バッチ境界の改行を保持する。
  • 追記のみ・置き換えなし。sentTail（送信済み末尾300字）を【文脈】としてAIに渡し、つながりを判断させる。よって人の同時編集と競合しない。
  • 録音停止後など 12秒 バッファ更新がなければ（stale）、ライブ末尾も含め全量を確定する。

6. そとづけの区切り情報と未確定表示

未確定テキストに \n を埋め込むと ProseMirror では空白に潰れる。そこで区切り位置を別チャネルで持つ：

  • サーバ statusMap: pendingText（緑+グレーの連結）／pendingGreenLen（緑＝先頭rawBuffer長）／pendingBreaks（緑内の改行オフセット配列・40字基準）。
  • 校正画面: 未確定インライン Decoration の <span> を、pendingBreaks の位置に <br> を入れて複数行描画。先頭 pendingGreenLen 文字を緑(#3fa874)、以降をグレー(#9ca3af)。
  • 認識画面: 録音者が送った生テキスト累積 sentRaw を、pendingGreenLen で「黒＝先頭(sentRaw.length−greenLen)／緑＝末尾greenLen分／グレー＝interim」に色分け。確定境界が校正画面と一致。

8. 校正画面の本文スクロール

  • 本文は h-[60vh] overflow-y-auto のスクロールボックス。新規追記に自動追従して末尾を表示。
  • 本文クリック／上スクロールで自動追従を停止。停止中に末尾が伸びたら右下に ▼ ボタンを表示し、押すと最下部へ移動して追従を再開（最下部到達でも自動再開）。

9. フェイルセーフ

  • 自動校正OFF時は rawBuffer を使わず、確定テキストを直接ドキュメントへ書く（interimのみグレー）。
  • バッファ過大（8000字超）時は未校正のまま一部書き出し（テキスト喪失防止）。切断時もバッファをflush。
  • オンデバイス認識が無音で停止しても onend で自動再起動。非対応ブラウザでは機能が静かに無効化。
